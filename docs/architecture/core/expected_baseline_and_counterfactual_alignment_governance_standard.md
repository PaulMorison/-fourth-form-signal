## Why This Standard Exists

The platform’s compounding edge depends not only on producing outcomes, outputs, simulations, post-mortem judgments, and learning signals, but also on disciplined control over the reference states by which those controlled layers are compared. Reference-state meaning sits between decision-time expectation on one side and later realised, simulated, counterfactual, baseline, and actual comparison on the other. If reference-state meaning drifts quietly, the stack begins to compare cleanly on logic whose authority is weaker than it looks.

Reference stability is too weak by default. A baseline can keep the same name and lose the same meaning. An expected state can keep the same label and still stop representing the same claim. A simulated state can remain numerically tidy and still cease to support the same comparison. A counterfactual can stay persuasive and still fail legitimate reuse. An actual state can still look concrete and still fail to function as a valid comparison anchor. If the platform cannot state what a governed expected state means, what a governed baseline state means, what a governed recommended state means, what a governed simulated state means, what a governed counterfactual state means, what a governed actual state means, what makes alignment legitimate, what makes baseline selection legitimate, what makes expected-to-actual comparability or simulated-to-realised comparability legitimate, and how later runs, reviewers, monitors, explainers, and learning gates may compare or reuse those reference states safely, then downstream trust weakens even while the comparison surfaces still look orderly.

The platform therefore needs one shared standard so that reference states accumulate as governed capital rather than as a pile of locally useful but semantically unstable baselines, expected-state snapshots, recommended-state summaries, simulation reference frames, counterfactual notes, and actual-state comparisons.

## Governed Reference-State Classes

This standard governs the shared semantic control layer that sits between already-controlled simulation, output, execution, outcome, and comparison artifacts on one side and trusted reusable reference-state meaning on the other.

### Local comparison reference

Local comparison reference is a locally useful expected, baseline, recommended, simulated, counterfactual, or actual state representation that may help one notebook, one review surface, one monitor, or one post-mortem preparation step without thereby qualifying for governed canonical status.

### Governed expected state

governed expected state is a governed reference-state class whose identity, scope, role, comparison posture, and lineage are explicit enough that later users can tell what future or projected state was expected and what expectation claim was not being made.

### Governed baseline state

governed baseline state is a governed reference-state class whose identity, scope, selection basis, role, comparison posture, and lineage are explicit enough that later users can tell why it serves as the baseline and why another state does not.

### Governed recommended state

governed recommended state is a governed reference-state class whose identity, scope, role as recommended comparison anchor, and lineage are explicit enough that later users can tell what state the recommendation implied for comparison use without confusing that reference-state meaning with output meaning.

### Governed simulated state

governed simulated state is a governed reference-state class whose identity, scope, synthetic basis, comparison role, and lineage are explicit enough that later users can tell what simulated condition is being used as a reference and what synthetic claim it does not settle.

### Governed counterfactual state

governed counterfactual state is a governed reference-state class whose identity, scope, counterfactual basis, comparison role, reuse boundary, and lineage are explicit enough that later users can tell what counterfactual reference is being claimed and where that claim remains bounded.

### Governed actual state

governed actual state is a governed reference-state class whose identity, scope, realised comparison role, and lineage are explicit enough that later users can tell what actual state is being used for reference comparison without collapsing that reference-state meaning into realised-outcome meaning.

### Canonical reference-state definition

canonical reference-state definition is the authoritative governed definition that states what a governed expected state, governed baseline state, governed recommended state, governed simulated state, governed counterfactual state, or governed actual state means, what semantic scope it applies to, what comparison role it supports, what boundaries constrain its reuse, and what semantic conditions must remain true for reuse to stay legitimate.

## Reference-State Identity and Semantic Scope

### Reference-state identity

reference-state identity is the stable identity linking one governed reference state to its canonical reference-state definition, named role, named scope, named comparison posture, named lifecycle posture, and later lineage rather than reducing it to a chart label, scenario alias, dashboard column, or local comparison handle.

### Reference-state semantic scope

reference-state semantic scope is the explicit statement of what business meaning, decision-loop meaning, population meaning, comparison meaning, and control boundary a governed reference state applies to and where that meaning must not be stretched by analogy or convenience.

### Reference-state legitimacy

reference-state legitimacy is the governed condition in which a governed reference state has stable identity, named role, named scope, named comparison posture, named lifecycle posture, and reconstructible lineage strong enough that later users can tell what reference claim it expresses and what claim it does not express.

Reference states must have named scope, named role, and named lineage. A governed reference state that cannot state what comparison job it performs, what case or case family it belongs to, what business boundary it applies to, what other states it may legitimately align with, what downstream uses it may support, and what downstream uses it does not authorize is too weak for canonical reuse.

## Expected, Baseline, and Recommended State Legitimacy

### Baseline legitimacy

baseline legitimacy is the governed condition in which a governed baseline state has explicit selection basis, explicit scope, explicit comparison role, explicit exclusion posture where relevant, and reconstructible lineage strong enough that later users can tell why it is the baseline and why another candidate state is not.

### Expected-state legitimacy

expected-state legitimacy is the governed condition in which a governed expected state has explicit expectation basis, explicit scope, explicit relation to the decision episode, and reconstructible lineage strong enough that later users can tell what expectation is being preserved and what expectation is not being claimed.

### Recommended-state legitimacy

recommended-state legitimacy is the governed condition in which a governed recommended state has explicit relation to recommendation context, explicit comparison role, explicit scope, and reconstructible lineage strong enough that later users can tell how the recommended state functions as a reference state without confusing that role with recommendation meaning itself.

an expected state is not the same thing as a baseline by itself. Expected-state meaning may inform baseline choice or later comparison, but it does not automatically become the valid baseline merely because it came earlier in time.

local usefulness is not the same thing as canonical baseline admission. One local baseline may feel analytically convenient and still fail governed legitimacy if its selection basis, scope, lineage, or comparison role are too unstable, too local, or too weakly governed for serious shared reuse.

reference-state presence is not the same thing as reference-state legitimacy. A visible expected state, baseline, or recommended state may still fail legitimacy if its role drifted, its scope blurred, its lineage weakened, or its selection basis became narrative rather than governed.

## Simulated, Counterfactual, and Actual State Legitimacy

### Counterfactual legitimacy

counterfactual legitimacy is the governed condition in which a governed counterfactual state has explicit counterfactual basis, explicit scope, explicit comparison role, explicit reuse boundary, and reconstructible lineage strong enough that later users can tell what counterfactual claim is being made and where it remains bounded.

### Simulated-state legitimacy

simulated-state legitimacy is the governed condition in which a governed simulated state has explicit synthetic basis, explicit scenario relation where relevant, explicit scope, and reconstructible lineage strong enough that later users can tell what simulated reference is being used and what it does not prove.

### Actual-state legitimacy

actual-state legitimacy is the governed condition in which a governed actual state has explicit realised comparison basis, explicit scope, explicit relation to the decision episode, and reconstructible lineage strong enough that later users can tell what actual reference is being preserved for comparison and what realised-outcome meaning remains owned elsewhere.

a simulated state is not the same thing as a counterfactual by itself. A simulated state may be one part of a counterfactual relation, but counterfactual meaning depends on the governed comparison role, not merely on synthetic generation.

an actual outcome is not the same thing as a valid baseline by itself. A realised result may later inform comparison, interpretation, or review and still fail to function as the legitimate baseline for a different comparison claim.

counterfactual reused outside legitimacy boundary is unacceptable. Counterfactual reuse must remain explicit, lineage-safe, and bounded strongly enough that later users can tell why the counterfactual still belongs and where it must not transfer.

## Alignment and Comparability Discipline

### Alignment legitimacy

alignment legitimacy is the governed condition in which expected, baseline, recommended, simulated, counterfactual, and actual states preserve explicit roles, explicit scope relations, explicit comparison basis, and explicit lineage strong enough that later users can tell why the reference-state alignment is legitimate rather than improvised.

### Expected-to-actual comparability

expected-to-actual comparability is the governed condition in which a governed expected state and a governed actual state preserve explicit role relation, explicit scope relation, explicit chronology relation, explicit observation relation where relevant, and explicit lineage strong enough that later users can tell why the comparison is legitimate rather than narrated after the fact.

### Simulated-to-realised comparability

simulated-to-realised comparability is the governed condition in which a governed simulated state and the realised comparison basis linked to a governed actual state preserve explicit scope relation, explicit scenario or simulation relation, explicit observation relation where relevant, and explicit lineage strong enough that later users can tell why the comparison is legitimate rather than superficially persuasive.

### Reference-state lineage

reference-state lineage is the reconstructible chain linking reference-state identity, canonical reference-state definition, role, scope, baseline-selection posture where relevant, counterfactual boundary where relevant, inherited or extended status, lifecycle status, invalidation or supersession where relevant, and later downstream use.

### Comparability-safe reference pair

comparability-safe reference pair is a pair of governed reference states whose semantic scope, role relation, lifecycle posture, baseline-selection relation, counterfactual boundary where relevant, and lineage remain explicit enough that comparison is legitimate rather than inferred.

### Non-comparable reference pair

non-comparable reference pair is a pair of governed reference states whose semantic scope, role relation, lifecycle posture, chronology relation, observation relation, or lineage differ materially enough that comparison would mislead later users even if the states share names, similar numbers, or superficially similar summaries.

### Inherited reference-state class

inherited reference-state class is a governed reference-state class reused without material semantic change from an earlier legitimate governed reference-state class whose identity and lineage remain explicit.

### Domain-extended reference-state class

domain-extended reference-state class is a governed reference-state class that extends an inherited reference-state class for a bounded domain need while keeping the extension explicit enough that comparability and semantic review remain possible.

### Promotion-safe reference use

promotion-safe reference use is the governed condition in which a governed reference state may be reused in a broader domain, broader review surface, broader explanation surface, broader monitoring surface, or policy-preparatory context without hiding local extension, reference-state drift, or comparability breakage under stable names.

comparability is not the same thing as superficial reference similarity. Shared labels, similar percentages, similar scenario names, similar output shapes, or similar charts do not by themselves make two governed reference states comparable.

comparison visibility is not the same thing as alignment legitimacy. A visible table, dashboard, or slide may still fail alignment legitimacy if the reference roles are unclear, the baseline selection drifted, the counterfactual boundary widened silently, or the actual comparison basis is too weak.

non-comparable reference pairs treated as equivalent is unacceptable. Alignment discipline requires the platform to preserve when reference-state comparison is legitimate and when it must remain blocked.

## Lifecycle Status

### Reference-state drift

reference-state drift is the material change in reference meaning, role, baseline-selection posture, counterfactual boundary, comparison posture, or scope that can hide underneath stable labels, stable dashboards, or stable scenario names.

### Superseded reference-state class

superseded reference-state class is a previously legitimate governed reference-state class whose governing role has been replaced by a newer governed reference-state class while preserving reconstructible historical identity and lineage.

### Deprecated reference-state class

deprecated reference-state class is a reference-state class that remains historically recognized but is no longer approved for new ordinary use except for bounded compatibility, historical interpretation, or controlled transition purposes.

### Retired reference-state class

retired reference-state class is a reference-state class removed from ordinary active use whose historical meaning remains preserved for lineage, audit, comparison, or retrospective interpretation.

### Invalidated reference-state class

invalidated reference-state class is a reference-state class whose meaning, lineage, comparison posture, or surrounding control integrity has broken badly enough that serious governed reuse is no longer permitted.

### Reference-state audit trace

reference-state audit trace is the reconstructible trace linking reference-state definitions, role changes, baseline-selection changes, comparison-basis changes, counterfactual-boundary changes, inheritance or extension, invalidation, supersession, and later downstream use.

reused baseline name with changed meaning is unacceptable. reference drift hidden under stable labels is unacceptable. invalidated reference state still used as current is unacceptable. Stable names and stable comparison surfaces do not guarantee stable meaning. Reference-state drift must remain explicit and auditable.

This standard prevents reused baseline name with changed meaning by requiring stable reference-state identity, explicit baseline legitimacy, and lineage-safe baseline changes.

This standard prevents expected state confused with recommendation by requiring that expected-state legitimacy remain separate from output meaning and recommendation meaning.

This standard prevents simulated state confused with actual state by requiring explicit simulated-state legitimacy, explicit actual-state legitimacy, and explicit simulated-to-realised comparability discipline.

This standard prevents counterfactual reused outside legitimacy boundary by requiring explicit counterfactual legitimacy, explicit reuse boundary, and promotion-safe reference use.

This standard prevents non-comparable reference pairs treated as equivalent by requiring comparability-safe reference pairs and explicit non-comparable reference pairs.

This standard prevents reference drift hidden under stable labels by requiring explicit reference-state drift visibility, reference-state audit trace, and reviewable lifecycle posture.

This standard prevents invalidated reference state still used as current by requiring explicit lifecycle posture, historical visibility, and block posture for invalid reuse.

This standard prevents local usefulness mistaken for canonical legitimacy by requiring that canonical baseline admission remain stricter than local convenience.

This standard prevents lineage break between reference state and realised outcome by requiring explicit reference-state lineage and explicit comparison discipline rather than retrospective narration.

This standard prevents silent mutation of reference interpretation by requiring explicit role discipline, explicit canonical reference-state definitions, and audit-ready change history.

## Implementation Expectations

Every governed expected state, governed baseline state, governed recommended state, governed simulated state, governed counterfactual state, and governed actual state must preserve a canonical reference-state definition, stable identity, explicit semantic scope, explicit role, intended audience where relevant, named lifecycle posture, and lineage strong enough for later reconstruction.

Every material baseline-selection change, comparison-basis change, role change, counterfactual-boundary change, actual-state interpretation change, inherited-versus-domain-extended change, lifecycle-status change, or cross-domain reuse change must remain audit-ready and reviewable rather than being treated as harmless analytical maintenance.

Every comparison service, review surface, explanation surface, monitoring surface, post-mortem-preparation surface, simulation interpretation surface, or learning-preparatory surface that can materially shape reference-state meaning must preserve enough traceability that later users can tell what states were aligned, what baseline was selected, what comparison relation was claimed, what lifecycle status applied, what reuse restrictions applied, and what changes materially affected meaning.

Reference states that are exploratory only, quarantined, compatibility-only, invalidated, superseded for ordinary use, or historically preserved but non-current must not be silently treated as current canonical baselines through dashboard defaults, search ranking, copied charts, repeated slide reuse, or convenience APIs. Where reference-state legitimacy is incomplete or restricted, that restriction itself must remain explicit rather than being laundered into normal comparison use.

Human review must remain available where baseline selection, counterfactual reuse, cross-domain transfer, or alignment-basis changes materially alter shared comparability, post-mortem preparation, monitoring interpretation, or learning-preparatory interpretation even when the chart or table change looks small.

## Governance and Approval

Consequential reference-state changes must follow the canon change-control and quality-gate standard, the canon navigation and reading-order standard, the end-to-end decision lifecycle composition standard, and the platform governance roles and approval authority matrix rather than entering the canon by convenience. Approval authority does not arise from analytical familiarity, dashboard ownership, simulation fluency, or local preference for one baseline. Where reference-state changes materially alter shared platform meaning, shared comparability, or cross-domain reuse posture, the relevant architecture and governance authorities must remain able to review and approve those changes explicitly.

Canonical reference-state admission, baseline-rule admission, counterfactual reuse admission, supersession, deprecation, retirement, invalidation, and major comparison-basis changes must preserve explicit approval lineage strong enough that later contributors can tell who approved the change, under what governance role, and why the change was treated as legitimate rather than merely convenient.

## Closing Position

Expected baseline and counterfactual alignment governance must remain a first-class platform control whose reference-state meaning, baseline-selection legitimacy, alignment legitimacy, comparability posture, counterfactual reuse boundary, lifecycle posture, and drift visibility remain explicit enough that the platform can compare seriously without mistaking visible charts, visible scenarios, visible baselines, or visible actuals for governed legitimacy. Reference states are powerful precisely because they feel intuitive and reusable. That apparent familiarity makes silent baseline laundering, silent reference drift, silent counterfactual overreach, and silent retrospective reinterpretation especially dangerous. The platform must therefore govern baseline-and-counterfactual meaning directly, preserve comparison discipline directly, and refuse uncontrolled baseline sprawl even when local convenience argues otherwise.
