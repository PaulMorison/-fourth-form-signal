# Canon-to-Implementation Mapping

## Purpose

This document turns the live core canon into a coding-ready implementation map. It treats the core canon in `docs/architecture/core` as the source of truth and maps each core document to the code modules, registries, persisted artifacts, operator surfaces, validations, runtime checks, and audit outputs required to implement it.

## Live Repo Baseline

- Core canon documents remain the source of truth.
- The live repo now contains executable source under `src/`, registry JSON under `registries/control_plane/`, contract schemas under `schemas/control_plane/`, focused unit tests under `tests/unit/`, and bootstrap validation under `src/bootstrap/first_control_plane_bootstrap.py`.
- The verified runtime/release boundary currently runs through `src/runtime/release/release_audit_trace.py`.
- This refresh treats the runtime/release inventory below as authoritative; broader document-wide counts should be refreshed separately if needed.

Status tags used in the module column:

- `[A]` already implemented in the live repo
- `[P]` partially implemented in the live repo
- `[M]` missing from the live repo
- `[U]` present but implementation role cannot be proven from the live repo

## Implementation Reading Order

Use this mapping in the following engineering order:

1. Platform registries, contracts, and audit spine
2. Shared decision-case, state-transition, and lifecycle orchestration modules
3. Raw ingestion, canonicalization, feature-definition, and dataset/version modules
4. Training, evaluation, model-run, artifact, and monitoring modules
5. Threshold, router, playbook, objective, and policy-output modules
6. Outcome capture, decision memory, and policy-learning modules
7. Operator surfaces and release/runtime/security layers

Anything that skips the registry, contract, and audit layers will violate the canon before the first business workflow is shipped.
