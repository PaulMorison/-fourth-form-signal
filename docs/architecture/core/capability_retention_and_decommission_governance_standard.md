# Capability Retention and Decommission Governance Standard for the Fourth Form Retail Decision Intelligence Platform

## Purpose

This document defines the shared platform standard for governed capability identity, governed capability semantic scope, governed active capability status, governed restricted capability status, governed compatibility-only capability status, governed frozen capability status, governed deprecated capability status, governed retired capability status, governed decommissioned capability status, retention legitimacy, restriction legitimacy, compatibility-only legitimacy, freeze legitimacy, deprecation legitimacy, retirement legitimacy, decommission legitimacy, reactivation legitimacy, capability lineage, capability drift, comparability-safe capability-state reuse, and promotion-safe capability use across all current and future domains.

It exists because the platform now has governed standards for commercial value realisation, release readiness, research and experimentation, post-deployment monitoring, outcome capture, objective governance, authority delegation, human review and escalation, canon change control, lifecycle composition, review resolution, exception and failure handling, reopen and reinstatement handling, and governance approval authority, but it still lacks one shared rule for when a governed capability remains worth keeping, when it must narrow, when it may survive only for compatibility, when it must freeze, when it becomes deprecated, when it is retired, when it is decommissioned, and when any return to broader use is legitimate rather than narrative convenience.

Without such a rule, the platform will drift into weak-value capabilities staying alive because they still exist, restricted capabilities being treated as fully active by habit, compatibility-only paths becoming covert strategic dependencies, frozen capabilities changing underneath stable labels, deprecated capabilities continuing to attract new ordinary use, retired capabilities being quietly brought back without legitimacy review, decommissioned capabilities still being assumed present by downstream consumers, and adjacent release, monitoring, review, or failure signals being mistaken for capability-lifecycle authority even when they do not own that meaning.

This document is therefore a control document for capability retention and decommission governance.

It is the canonical capability retention and decommission governance standard for the platform. Future governed capabilities, capability registries, lifecycle-state changes, compatibility surfaces, freeze decisions, retirement decisions, decommission decisions, reactivation proposals, and domain-local capability extensions must align with it when preserving governed active capability, governed restricted capability, governed compatibility-only capability, governed frozen capability, governed deprecated capability, governed retired capability, governed decommissioned capability, canonical capability definition, capability identity, capability semantic scope, retention legitimacy, restriction legitimacy, compatibility-only legitimacy, freeze legitimacy, deprecation legitimacy, retirement legitimacy, decommission legitimacy, reactivation legitimacy, capability lineage, capability drift, inherited capability class, domain-extended capability class, superseded capability class, invalidated capability class, comparability-safe capability-state pair, non-comparable capability-state pair, promotion-safe capability use, and capability audit trace unless a formal decision record explicitly revises it.

## Why This Standard Exists

The platform's compounding edge depends not only on building capabilities, releasing them, monitoring them, reviewing them, and learning from them, but also on disciplined control over whether those capabilities should still exist in their current operational posture. Capability lifecycle-state meaning sits between technical availability on one side and trusted governed reuse on the other. If lifecycle-state meaning drifts quietly, the platform begins to keep, narrow, freeze, retire, or remove capabilities on logic whose authority is weaker than it looks.

Lifecycle stability is too weak by default. A capability can keep the same name and lose the same justification. A restricted capability can still look available and still stop being safe for broad use. A compatibility-only path can still look stable and still become an illegitimate new dependency. A frozen capability can still seem unchanged and still drift through hidden semantics, hidden interfaces, or hidden downstream assumptions. A retired capability can still appear in memory and still fail any legitimate return path. A decommissioned capability can still leave traces behind and still be wrongly treated as current. If the platform cannot state what a governed active capability means, what a governed restricted capability means, what a governed compatibility-only capability means, what a governed frozen capability means, what a governed deprecated capability means, what a governed retired capability means, what a governed decommissioned capability means, and what makes retention, restriction, compatibility-only support, freeze, retirement, decommission, or reactivation legitimate, then downstream trust weakens even while the capability surface still looks orderly.

The platform therefore needs one shared standard so that capabilities accumulate as governed capital rather than as a pile of locally useful but semantically unstable services, workflows, automations, dashboards, monitoring paths, compatibility shims, fallbacks, and legacy surfaces.

An active capability may still be weakly justified if its value pathway collapsed, its authority basis narrowed, or its lifecycle state simply failed to update. A restricted capability may be semantically legitimate when bounded use remains appropriate and still must not be treated as platform-wide ordinary availability. Compatibility-only support may be necessary for safe migration and still must not become the excuse for indefinite strategic dependence. A frozen capability may remain current for bounded use while no longer being open to extension. Retirement preserves historical meaning and removes ordinary use. Decommission removes operational entitlement and downstream assumption more strongly than retirement does. Reactivation demands stricter proof than casual return.

## Governed Capability Lifecycle Classes

This standard governs the shared lifecycle-state control layer that sits between already-existing capabilities on one side and trusted governed retention, narrowing, retirement, or removal meaning on the other.

### Governed active capability

governed active capability is a governed capability whose identity, semantic scope, ordinary supported use, lifecycle posture, and lineage remain explicit enough that later users can tell why the capability is still approved for current serious use and what use it does not authorize.

### Governed restricted capability

governed restricted capability is a governed capability whose continued use is bounded by explicit scope, audience, operational condition, risk posture, dependency posture, or authority condition strong enough that later users can tell where the capability still belongs and where it no longer does.

### Governed compatibility-only capability

governed compatibility-only capability is a governed capability preserved only to support bounded backward compatibility, controlled migration, historical replay, or explicitly governed consumer continuity strongly enough that later users can tell why it still exists and why new ordinary dependence is not legitimate.

### Governed frozen capability

governed frozen capability is a governed capability whose current meaning and bounded current use remain preserved while semantic expansion, interface expansion, ordinary enhancement, or quiet redesign are blocked strongly enough that later users can tell why the capability is frozen and what change paths are no longer legitimate.

### Governed deprecated capability

governed deprecated capability is a governed capability that remains historically recognized and may remain available in bounded contexts but is no longer approved for new ordinary use except for explicitly governed transition or compatibility purposes.

### Governed retired capability

governed retired capability is a governed capability removed from ordinary active use whose historical meaning, lineage, and prior role remain preserved for audit, interpretation, comparison, or bounded historical understanding.

### Governed decommissioned capability

governed decommissioned capability is a governed capability whose operational entitlement, ordinary execution posture, and downstream assumption have been removed strongly enough that remaining traces do not imply current availability, current safety, or current legitimacy.

### Canonical capability definition

canonical capability definition is the authoritative governed definition that states what a capability means, what semantic scope it applies to, what role it serves, what lifecycle states it may legitimately occupy, what boundaries constrain its use, and what semantic conditions must remain true for retention, restriction, retirement, decommission, or reactivation to stay legitimate.

## Capability Identity and Semantic Scope

### Capability identity

capability identity is the stable identity linking one governed capability to its canonical capability definition, named role, named semantic scope, named lifecycle posture, dependency posture where relevant, and later lineage rather than reducing it to a service name, feature label, dashboard tab, or local operational nickname.

### Capability semantic scope

capability semantic scope is the explicit statement of what business meaning, decision-loop meaning, operational meaning, audience meaning, dependency meaning, and control boundary a governed capability applies to and where that meaning must not be stretched by analogy or convenience.

Capabilities must have named scope, named purpose, and named lifecycle posture. A governed capability that cannot state what work it serves, what surfaces it reaches, what dependencies it carries or supports, what conditions still justify its existence, and what conditions now constrain it is too weak for canonical reuse.

## Active, Restricted, and Compatibility-Only Capability Legitimacy

### Retention legitimacy

retention legitimacy is the governed condition in which a capability has explicit value basis, explicit scope, explicit authority basis, explicit dependency posture, explicit lifecycle posture, and reconstructible lineage strong enough that later users can tell why it still deserves to exist as a governed capability rather than as tolerated residue.

### Restriction legitimacy

restriction legitimacy is the governed condition in which a capability may remain in use only under explicit narrowing of scope, audience, operational boundary, dependency boundary, authority condition, or exposure boundary strong enough that later users can tell why the restriction exists and why broader use is not legitimate.

### Compatibility-only legitimacy

compatibility-only legitimacy is the governed condition in which a capability remains preserved only for bounded migration, bounded consumer continuity, bounded replay, or other explicitly transitional purposes strong enough that later users can tell why the capability still exists and why new ordinary strategic dependence is illegitimate.

an active capability is not the same thing as a justified capability by itself. A capability may still be active in operations and still fail retention legitimacy if value evidence weakened, authority support narrowed, dependency burden expanded, or lifecycle posture stopped reflecting reality.

a restricted capability is not the same thing as a failed capability by itself. A capability may remain legitimate under restriction when bounded use still protects value, safety, compatibility, or continuity better than immediate retirement or decommission would.

compatibility-only support is not the same thing as active strategic retention by itself. Compatibility-only support exists to preserve controlled transition or bounded dependency continuity, not to keep a capability appearing more current or more justified than it really is.

retention visibility is not the same thing as retention legitimacy. A visible capability in a registry, interface surface, or operational path may still fail retention legitimacy if its basis for continued existence is too weak, too local, too stale, or too poorly governed.

## Frozen, Deprecated, Retired, Decommission, and Reactivation Legitimacy

### Freeze legitimacy

freeze legitimacy is the governed condition in which a capability may remain present for bounded use while enhancement, semantic expansion, or ordinary design change is blocked explicitly enough that later users can tell why the capability is frozen and what changes are no longer legitimate.

### Deprecation legitimacy

deprecation legitimacy is the governed condition in which a capability remains historically recognized and may remain available in bounded transition paths while new ordinary use is explicitly discouraged or prohibited strongly enough that later users can tell the capability is weaker than active or restricted status.

### Retirement legitimacy

retirement legitimacy is the governed condition in which a capability is removed from ordinary active use while preserving historical meaning, lineage, and bounded retrospective interpretability strongly enough that later users can tell why it is retired and why current ordinary use is no longer legitimate.

### Decommission legitimacy

decommission legitimacy is the governed condition in which a capability's operational entitlement, ordinary execution posture, and downstream assumption have been removed strongly enough that serious current reuse is no longer permitted and downstream consumers can tell that present availability must not be assumed.

### Reactivation legitimacy

reactivation legitimacy is the governed condition in which a previously restricted, frozen, deprecated, or retired capability may return to a stronger governed lifecycle state only when value basis, authority basis, dependency posture, monitoring posture where relevant, and lineage remain explicit enough that later users can tell why the return is legitimate rather than nostalgic or convenient.

a frozen capability is not the same thing as a deprecated capability by itself. Freeze blocks ordinary semantic change while allowing bounded current use to continue. Deprecation makes new ordinary use weaker or illegitimate even if the capability still exists.

retirement is not the same thing as decommission by itself. Retirement preserves historical meaning of a capability that no longer belongs in ordinary current use. Decommission removes operational entitlement and downstream assumption strongly enough that any later return is not casual reuse and may require explicit reactivation review or even a new governed capability admission path.

local usefulness is not the same thing as canonical capability retention. One local team may still find a capability helpful while the platform-level basis for keeping it active, broad, or current has already become too weak.

capability presence is not the same thing as capability legitimacy. A capability can still exist in runtime, storage, interfaces, dashboards, or operator habit and still fail active, restricted, compatibility-only, frozen, deprecated, retired, or decommission legitimacy if the governing posture is no longer explicit.

## Retention, Restriction, and Decommission Discipline

### Capability lineage

capability lineage is the reconstructible chain linking capability identity, canonical capability definition, lifecycle-state changes, dependency posture changes, boundary changes, authority conditions where relevant, inherited or extended status, invalidation or supersession where relevant, and later downstream use.

### Comparability-safe capability-state pair

comparability-safe capability-state pair is a pair of governed capability states whose capability identity, semantic scope, lifecycle meaning, dependency posture, authority posture where relevant, and lineage remain explicit enough that comparison is legitimate rather than narrated after the fact.

### Non-comparable capability-state pair

non-comparable capability-state pair is a pair of governed capability states whose semantic scope, lifecycle meaning, dependency posture, authority posture, or lineage differ materially enough that comparison would mislead later users even if the capability names remain stable.

### Promotion-safe capability use

promotion-safe capability use is the governed condition in which a capability may be preserved, broadened, restricted, frozen, retired, or otherwise reused in a broader domain, broader release surface, broader dependency surface, or broader governance context without hiding lifecycle-state drift under stable labels.

Retention discipline requires the platform to preserve why a capability remains active, why it is restricted rather than retired, why it is compatibility-only rather than active, why it is frozen rather than evolving, and why retirement or decommission is or is not legitimate. Restriction discipline requires the platform to preserve what narrowing actually means and what downstream use is now outside bounds. Decommission discipline requires the platform to preserve what operational assumption has been removed, what migration or cleanup obligations remain, and what historical traces must stay visible without implying current legitimacy.

Comparability is not the same thing as superficial capability similarity. Stable names, similar endpoints, similar dashboards, similar operational tasks, or similar migration stories do not by themselves make one capability state comparable to another.

## Lifecycle Status

### Capability drift

capability drift is the material change in capability meaning, lifecycle posture, dependency posture, allowed audience, or operational boundary that can hide underneath stable labels, stable interfaces, stable dashboards, or stable runtime presence.

### Inherited capability class

inherited capability class is a governed capability class reused without material semantic change from an earlier legitimate governed capability class whose identity and lineage remain explicit.

### Domain-extended capability class

domain-extended capability class is a governed capability class that extends an inherited capability class for a bounded domain need while keeping the extension explicit enough that lifecycle review and comparability remain possible.

### Superseded capability class

superseded capability class is a previously legitimate governed capability class whose governing role has been replaced by a newer governed capability class while preserving reconstructible historical identity and lineage.

### Invalidated capability class

invalidated capability class is a governed capability class whose meaning, lifecycle posture, dependency posture, or surrounding control integrity has broken badly enough that serious governed reuse is no longer permitted.

### Capability audit trace

capability audit trace is the reconstructible trace linking capability definitions, lifecycle-state changes, restriction changes, compatibility-only changes, freeze changes, retirement decisions, decommission decisions, reactivation decisions, inheritance or extension, invalidation, supersession, and later downstream use.

Stable capability labels do not guarantee stable legitimacy. capability drift must remain explicit and auditable. Inherited capability classes must remain distinguishable from domain-extended capability classes. Superseded capability classes must remain historically identifiable. Invalidated capability classes must remain explicitly unusable for serious governed reuse even if old tooling still remembers them.

## Failure Modes This Standard Prevents

This standard prevents active capability kept alive after value collapse by requiring explicit retention legitimacy rather than treating runtime presence as justification.

This standard prevents restricted capability quietly treated as fully active by requiring explicit restriction legitimacy, explicit restriction boundaries, and reviewable current-state visibility.

This standard prevents compatibility-only path used for new strategic dependence by requiring explicit compatibility-only legitimacy and by blocking new ordinary dependence from hiding inside transition posture.

This standard prevents frozen capability changed under stable name by requiring explicit freeze legitimacy, explicit change blocking, and lifecycle-state visibility.

This standard prevents deprecated capability still used for new ordinary work by requiring explicit deprecation legitimacy and by preserving that deprecation remains weaker than active or restricted use.

This standard prevents retired capability reintroduced without legitimate reactivation by requiring explicit reactivation legitimacy, preserved lineage, and stronger proof than local convenience.

This standard prevents decommissioned capability still assumed present by downstream consumers by requiring explicit decommission legitimacy, explicit removal of ordinary entitlement, and downstream-assumption visibility.

This standard prevents local usefulness mistaken for retention legitimacy by requiring that canonical capability retention remain stricter than local convenience.

This standard prevents failure-state handling confused with capability lifecycle state by requiring capability lifecycle posture to remain separate from degraded, blocked, quarantined, invalid, or recovered episode posture.

This standard prevents case reopen confused with capability reactivation by requiring capability lifecycle-state changes to remain separate from case-level reopen, revisit, or reinstatement handling.

## Canonical Control Statements

Capability lifecycle-state meaning must remain separate from release state even when release evidence materially influences whether a capability may remain active, restricted, frozen, or retired. Capability lifecycle-state meaning must remain separate from experiment state even when research produces the evidence that a capability should be broadened, narrowed, or removed. Capability lifecycle-state meaning must remain separate from monitoring alerts even when monitored degradation helps justify restriction, freeze, retirement, or decommission. Capability lifecycle-state meaning must remain separate from case-level review, failure, or reopen handling even when repeated cases reveal that the capability no longer deserves its former posture.

Retention legitimacy requires explicit current value basis, explicit authority basis, explicit dependency posture, and lineage strong enough for serious reuse. Restriction legitimacy requires explicit narrowing. Compatibility-only legitimacy requires bounded transition rather than broad current endorsement. Freeze legitimacy requires explicit blocking of semantic change. Deprecation legitimacy requires visible weakening of new-use approval. Retirement legitimacy requires explicit end of ordinary use with historical visibility preserved. Decommission legitimacy requires removal of operational entitlement and downstream assumption. Reactivation legitimacy requires stronger proof than nostalgia, sunk cost, or resumed convenience.

Capability state changes must remain auditable, lineage-safe, and reconstructible enough that later users can tell what changed, why it changed, who approved it, what downstream boundaries moved, and what consumers must now stop assuming. Silent lifecycle drift is unacceptable. Stable labels do not justify stable trust.

## Implementation Expectations

Every governed capability must preserve a canonical capability definition, stable identity, explicit semantic scope, explicit current lifecycle state, explicit dependency posture where relevant, and lineage strong enough for later reconstruction. Capability registries, service catalogs, dashboard inventories, workflow maps, automation registries, and other operational surfaces that refer to governed capabilities must expose lifecycle posture clearly enough that restricted, compatibility-only, frozen, deprecated, retired, and decommissioned are not silently flattened into generic availability.

State changes must preserve the change basis, effective boundary, affected consumers, migration implications where relevant, compatibility obligations where relevant, and the authority path that approved the change. Restriction must preserve what remains allowed and what is no longer allowed. Compatibility-only posture must preserve what consumers remain tolerated temporarily and what new dependence is prohibited. Freeze must preserve what kinds of change are blocked. Retirement must preserve what historical interpretation still matters. Decommission must preserve what entitlement and downstream assumption have been removed. Reactivation proposals must preserve what changed strongly enough to justify return and why ordinary new admission or rebuild is not the more truthful framing where applicable.

Where capabilities participate in release, monitoring, review, or failure-sensitive handling, those adjacent records must remain linkable without collapsing their meanings into lifecycle-state meaning. Implementation must preserve enough audit trace that later reviewers can reconstruct when a capability moved from active to restricted, from restricted to compatibility-only, from frozen to retired, or from retired to decommissioned, and why those changes were legitimate.

## Governance and Approval

Changes to shared capability lifecycle-state meaning, shared lifecycle-state classes, retention criteria, restriction posture, compatibility-only posture, freeze posture, retirement posture, decommission posture, or reactivation posture are consequential platform changes. They intersect commercial-value interpretation, release consequence, experiment consequence, monitoring consequence, authority consequence, lifecycle consequence, and downstream dependency consequence. Review and approval must therefore align with the governance authority matrix at the stricter applicable path, with Platform Owner, Architecture Authority, Commercial Authority, Governance and Boundary Authority, Implementation Authority, and affected Domain Authority involved where the change materially touches their control surfaces.

This standard should be treated as the controlling reference for when a capability remains governed current, when it is intentionally narrowed, when it survives only for compatibility, when it is frozen, when it is deprecated, when it is retired, when it is decommissioned, and when any stronger return is legitimate. Adjacent standards may contribute evidence or trigger conditions, but they do not redefine capability-lifecycle meaning by themselves.

## Non-Negotiables

1. Every governed capability must preserve stable identity, explicit semantic scope, and explicit current lifecycle posture, because capability presence is not the same thing as capability legitimacy.
2. Capabilities may remain active only when retention legitimacy remains explicit, because an active capability is not the same thing as a justified capability by itself.
3. Restricted capabilities must preserve explicit restriction boundaries and explicit allowed use, because a restricted capability is not the same thing as a failed capability by itself.
4. Compatibility-only capabilities must block new ordinary strategic dependence, because compatibility-only support is not the same thing as active strategic retention by itself.
5. Frozen capabilities must block silent semantic expansion under stable labels, because a frozen capability is not the same thing as a deprecated capability by itself.
6. Deprecated, retired, and decommissioned capabilities must remain distinguishable, because retirement is not the same thing as decommission by itself.
7. Reactivation must remain stricter than resumed convenience, because local usefulness is not the same thing as canonical capability retention.
8. Capability lifecycle-state meaning must remain separate from release, monitoring, review, failure, and case re-entry meaning, because adjacent evidence does not by itself own lifecycle legitimacy.
9. Capability drift, supersession, invalidation, and audit history must remain explicit, because retention visibility is not the same thing as retention legitimacy.
10. Future capability-retention-and-decommission extensions must be placed according to control role, not convenience.

## Closing Position

The platform cannot stay disciplined if capabilities are allowed to keep existing without a governed answer to whether they should still exist, how narrowly they still belong, what transitional support remains tolerated, what change is blocked, what use is now discouraged, what has already left ordinary use, what has been removed from operational trust, and what would justify any serious return. Capability lifecycle-state meaning must therefore remain explicit, lineage-safe, and harder to fake than technical presence. That is how the platform prevents legacy residue, migration theater, silent dependence, and nostalgic resurrection from impersonating governed capability stewardship.
